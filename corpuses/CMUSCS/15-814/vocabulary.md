# 15-814 — Types and Programming Languages · Vocabulary

Source: CMU 15-814 Types and Programming Languages, Fall 2024 (Jan Hoffmann, with Frank Pfenning). Lecture notes at https://www.cs.cmu.edu/~janh/courses/814/24/schedule/ . Terms are drawn from the publicly posted per-lecture notes (lectures 1–6, 9, 10/11, 14, 16, 19); intervening lectures were not posted, so topics such as product/sum types, recursive types, existentials, dynamic and gradual typing, subtyping, and references are outside the notes available here. Ordered by course progression. The course follows Harper's *Practical Foundations for Programming Languages* (PFPL).

**Programming language as a mathematical object** — The course's foundational stance: a language is not defined by a compiler, implementation, or informal standard, but formally by its static and dynamic semantics, usually given by rule induction.

**Static semantics** — The part of a language definition that specifies the set of well-formed programs (more generally, expressions), typically via typing rules.

**Dynamic semantics** — The part of a language definition that specifies the result of running (evaluating) a program.

**Abstraction and composition** — The two organizing themes of the study of languages: abstractions hide information and enforce guarantees, and composition combines programs without breaking those abstractions.

**Turing completeness** — The property that a language can implement the same set of (partial) functions as any other universal model; most practical languages share this expressivity.

**Type theory** — The systematic study of abstractions in which types classify different forms of data and computation.

**λ-calculus** — Church's pure calculus of functions, built from variables, λ-abstractions, and applications; used as the simplest setting in which to study computation.

**Church–Turing Thesis** — The claim that the effectively computable partial functions are exactly those implementable by Turing machines, equivalently in the λ-calculus.

**λ-abstraction** — The expression `λx. e`, denoting the function itself (rather than only its values on arguments); written `lam(x.e)` in abstract syntax.

**Application** — The expression `e1 e2` applying one expression to another; written `app(e1, e2)` in abstract syntax. Juxtaposition is left-associative.

**Concrete syntax** — The surface notation for a language, including conventions such as parenthesization and associativity used by an implementation.

**Abstract syntax / abstract binding trees (ABTs)** — A representation of syntax that generalizes abstract syntax trees by adding a built-in notion of variables and binding; operators (e.g. `lam`, `app`) each carry a fixed arity and binding structure.

**Operator** — A syntactic constructor of an ABT with a fixed number of arguments and a specified binding structure (e.g. `lam` binds one variable in its body).

**Sort** — A kind of syntactic object an ABT variable can range over (e.g. expressions, or in richer languages both expressions and types).

**Variable (expression variable)** — A placeholder standing for an expression that may be substituted for it; substitution gives variables their meaning.

**Substitution** — Replacement of a variable `x` by an expression `e′` in `e`, written `[e′/x]e`, defined so as to avoid variable capture.

**Binding and scope** — `λx. e` binds `x` with scope `e`; bound variables may be consistently renamed without changing meaning.

**Free variables** — Variables occurring in an expression that are not bound; defined by the judgment `x ∈ e`. Changing a free variable's name changes meaning.

**Closed expression** — An expression with no free variables.

**α-conversion / α-equivalence** — Consistent renaming of bound variables; expressions differing only in bound-variable names are α-equivalent, and are silently identified throughout the course.

**Variable capture** — The error in which a free variable becomes accidentally bound during naive substitution; avoided by renaming (α-conversion), giving a total substitution.

**β-reduction** — The directed computation step `(λx. e2) e1 ⟼ [e1/x]e2`; "the engine of computation" in the λ-calculus.

**β-conversion / β-equivalence** — The equational theory generated by β-reduction (together with α-conversion), reflexive, symmetric, and transitive; two expressions are β-equivalent if one converts to the other.

**Combinator** — A closed λ-expression (no free variables), such as the identity `I = λx. x`, the constant `K = λx.λy. x`, or composition `B = λf.λg.λx. f (g x)`.

**Normal form** — An expression that cannot be reduced further by β-reduction; used to represent the values (results) of computations.

**Nontermination (Ω)** — The phenomenon of computations without a normal form; the canonical example is `Ω = (λx. x x)(λx. x x)`, which β-reduces only to itself.

**Redex** — A reducible expression of the form `(λx. e) e′`, the site at which β-reduction applies.

**Leftmost-outermost / normal-order reduction** — The reduction strategy that always reduces the leftmost, outermost redex; it is complete, finding a normal form whenever one exists.

**Call-by-name / call-by-need / call-by-value** — Evaluation strategies: call-by-name (related to normal-order) passes arguments unevaluated; call-by-need is the Haskell variant sharing results; call-by-value (eager) reduces arguments before substituting.

**Confluence (Church–Rosser property)** — The property that any two reduction sequences from a common expression can be brought back together: if `e ⟼* e1` and `e ⟼* e2` then some `e′` has `e1 ⟼* e′` and `e2 ⟼* e′`.

**Church–Rosser Theorem** — The theorem establishing confluence for β-reduction; it implies uniqueness of normal forms.

**Uniqueness of normal forms** — The consequence of confluence that an expression has at most one normal form (up to renaming), so the result of a computation is well-defined.

**Church encoding** — Representation of data as closed normal forms in the pure λ-calculus; the canonical encoding corresponds to the datatype's elimination form.

**Booleans (Church)** — `true = λx.λy. x`, `false = λx.λy. y`; a Boolean applied to two arguments branches, exactly like a conditional.

**Elimination form** — The way a value of a type is consumed (e.g. the conditional `if-then-else` for Booleans; the recursor for natural numbers).

**Introduction form** — The way a value of a type is constructed (e.g. zero and successor for natural numbers).

**Church numerals** — Representation of `n` as `λs.λz. sⁿ z`, iterating its first argument `n` times over its second; addition, multiplication, etc. are defined by iteration.

**Extensionality** — The property (which the λ-calculus's data representations lack) that functions agreeing on all relevant arguments are equal; `not` and `not′` behave identically yet are distinct normal forms.

**Schema of iteration** — Recursion of the form `f 0 = c`, `f(n+1) = g (f n)`, where the step has access only to the previous result.

**Schema of primitive recursion** — Recursion of the form `f 0 = c`, `f(n+1) = h n (f n)`, where the step also has access to the predecessor `n`.

**Pairs (encoding) / continuation** — Pairs encoded functionally as `⟨e1,e2⟩ = λk. k e1 e2`, applying a continuation `k` to both components; used to define the predecessor by rebuilding `n` during iteration.

**General recursion** — Recursion `f = h f` allowing arbitrary recursive calls, more expressive than primitive recursion and permitting nontermination.

**Fixed point** — A function `f` with `f = h f`; general recursion asks for a fixed point of `h`.

**Y combinator** — The closed λ-expression `Y = λh. (λx. h (x x))(λx. h (x x))` satisfying `Y h =β h (Y h)`; it realizes general recursion and can diverge.

**Partial recursive function** — A function that may be undefined (have no normal form) on some inputs; representable via the Y combinator.

**Judgment** — An assertion about objects (e.g. `e exp`, `Γ ⊢ e : τ`); the basic unit inductively defined by inference rules.

**Inference rule** — A rule `J1 … Jn / J` whose premises `Ji` imply its conclusion `J`; rules are named for reference.

**Derivation / derivation tree** — A tree of inference-rule instances witnessing that a judgment is derivable; the inductive definition is the smallest set of judgments closed under the rules.

**Rule induction** — Proof principle for inductively defined judgments: assume the statement for a rule's premises (induction hypothesis) and prove it for the conclusion; equivalently, induction on derivation trees.

**Inversion** — Reasoning that, given a derivable judgment, determines which rules could have produced it and recovers their premises; packaged as inversion lemmas.

**Strong vs. weak reduction** — Strong reduction permits reducing under λ (rule allowing `λx. e =β λx. e′`); weak reduction (call-by-name/value) does not, so all abstractions are normal forms.

**System T (Gödel's T)** — A typed language with natural numbers, higher-order functions, and primitive recursion (the recursor); introduced by Gödel and here presented as a normalizing programming language.

**Curry–Howard Correspondence** — The link between mathematical logic and normalizing programming languages first encountered with System T (Gödel's response to Hilbert's program).

**Type** — A classifier of expressions; System T's types are `nat` and function types `τ1 → τ2`.

**Function type** — `τ1 → τ2`, the type of functions from `τ1` to `τ2`; introduced by λ-abstraction and eliminated by application.

**Recursor** — System T's elimination form for `nat`, `rec{e0; x.y.e1}(e)`, implementing primitive recursion; its arbitrary result type (including function types) makes non-primitive-recursive functions like Ackermann's definable.

**Numeral** — The System T representation of `n` built from `z` and `s(·)` (unary encoding).

**Typing judgment** — `Γ ⊢ e : τ`, asserting expression `e` has type `τ` in variable context `Γ`.

**Variable context (Γ)** — A finite map from variables to types (each variable at most once) recording the types of free variables; order is immaterial in the structural setting.

**Hypothetical judgment** — A judgment made under assumptions; `Γ ⊢ e : τ` is hypothetical, with the variable typings in `Γ` as hypotheses.

**Syntax-directed rules** — A rule set with exactly one rule per syntactic form, so the form of the expression determines the applicable rule; makes type checking straightforward.

**Substitution lemma** — If `Γ, x : τ′ ⊢ e : τ` and `Γ ⊢ e′ : τ′` then `Γ ⊢ [e′/x]e : τ`; substituting a well-typed expression preserves typing.

**Uniqueness of types** — In System T, a given expression has at most one type in a given context (aided by type annotations on abstractions).

**Structural properties** — Properties of contexts a type system may enjoy: weakening, contraction, and exchange.

**Weakening** — The admissibility of adding an unused variable to the context: if `Γ ⊢ e : τ` then `Γ, x : τ′ ⊢ e : τ`.

**Contraction** — The admissibility of using a variable more than once: from `Γ, x1:τ, x2:τ ⊢ e : τ′` derive `Γ, x:τ ⊢ [x,x/x1,x2]e : τ′`.

**Exchange** — The admissibility of reordering variables in the context without affecting typability.

**Operational semantics** — A style of dynamic semantics describing how expressions are evaluated (close to an interpreter), contrasted with denotational semantics.

**Structural dynamics / small-step semantics** — Operational semantics given as a transition system `e ⟼ e′` between (closed) expressions, with search rules and leaf (computation) rules.

**Transition system** — States plus a transition relation; here states are expressions and transitions are single evaluation steps.

**Value** — A final state of the transition system that does not step further; in System T, numerals and λ-abstractions.

**Multi-step evaluation** — The reflexive–transitive closure `e ⟼* e′` of the step relation.

**Determinacy** — The property that each non-value expression has exactly one transition (up to α-equivalence).

**Eager vs. lazy evaluation** — Call-by-value (eager) evaluates a function's argument to a value before substituting; call-by-name (lazy) substitutes the unevaluated argument, letting variables stand for general expressions.

**Ackermann's function** — A fast-growing total function, not primitive recursive, yet definable in System T thanks to the recursor's arbitrary (higher-order) result type.

**Type soundness** — The link between statics and dynamics establishing that well-typed programs do not get stuck, proved via progress and preservation.

**Progress** — If `e : τ` then either `e` is a value or `e ⟼ e′` for some `e′` (well-typed programs are not stuck).

**Preservation** — If `e : τ` and `e ⟼ e′` then `e′ : τ` (types are preserved by evaluation); together with progress yields "well-typed programs don't go wrong."

**Canonical-forms lemma** — A characterization of the possible values at each type (e.g. a value of type `nat` is a numeral, of a function type a λ-abstraction), used in soundness and normalization proofs.

**Normalization** — The property that every well-typed program evaluates to a value; System T is normalizing, i.e. a total language.

**Total language** — A language in which all programs terminate; contrasted with partial languages like PCF and the λ-calculus.

**Logical relations (Tait's method)** — A proof technique defining a predicate/relation by induction on type structure to prove properties (e.g. normalization) that plain induction on typing cannot; also called the method of candidates.

**Hereditary termination** — The specific logical relation used for System T normalization: `HTnat(e)` if `e` evaluates to a value, and `HTτ1→τ2(e)` if `e` evaluates to a function mapping hereditarily terminating arguments to hereditarily terminating results.

**Canonicity** — Normalization at the base type: every closed program `e : nat` evaluates to some numeral `n`.

**Definability / undefinability** — A function `f : Nᵏ → N` is definable in System T if some expression computes it; System T defines exactly the functions provably total in Peano arithmetic, but some total computable functions are undefinable.

**Self-interpreter / universal function** — A function `funiv(⌜e⌝, m)` that interprets an encoded program `e` on input `m`; System T cannot define its own self-interpreter, though the λ-calculus can implement a universal function for T.

**Gödel numbering** — An effective encoding `⌜e⌝ ∈ N` of expressions as numbers, from which the expression can be recovered.

**Diagonalization** — The proof technique (used for undefinability/incompleteness) of applying a program to its own encoding to derive a contradiction with normalization.

**Blum's Size Theorem** — The result that in any expressive total language there are functions whose shortest implementation is enormously larger than a λ-calculus implementation, because a total-language program embeds its own termination proof.

**Polymorphism** — The possibility of a single expression having multiple types.

**Ad hoc vs. parametric polymorphism** — Strachey's distinction: ad hoc polymorphism gives different implementations at different types (e.g. `plus` on ints vs. floats); parametric polymorphism uses one implementation behaving the same at all types (e.g. the identity function).

**System F / polymorphic λ-calculus** — Girard's and Reynolds's system for parametric polymorphism, adding universally quantified types.

**Universally quantified type** — `∀(t. τ)`, the type of a value that takes a type as an argument; e.g. the polymorphic identity has type `∀(t. t → t)`.

**Type abstraction and type application** — `Λ(t) e` abstracts over a type variable (capital lambda), and `e [τ]` instantiates a polymorphic expression at a type, substituting into the type: `e [τ′] : [τ′/t]τ`.

**Type variable / type context (∆)** — A variable `t` ranging over types, declared `t type` in a type context `∆`; the typing judgment becomes `∆ Γ ⊢ e : τ`.

**Impredicativity** — The self-referential character of System F's `∀`, which can be instantiated at any type including one built from itself (e.g. `u = ∀(t. t → t)` instantiated at `u`); a source of foundational concern but proved consistent by Girard.

**Denotational semantics** — A style of dynamic semantics assigning mathematical objects (e.g. mathematical functions) to expressions, contrasted with operational and with axiomatic and game semantics.

**Evaluation dynamics / big-step (natural) semantics** — An operational semantics inductively relating expressions to values, `e ⇓ v`, with no notion of individual step; close to an efficient recursive interpreter.

**Soundness and completeness of dynamics** — Two dynamic semantics are equivalent when each `e ⇓ v` corresponds to `e ⟼* v` (soundness) and conversely (completeness); proved for evaluation dynamics vs. structural dynamics and vs. the K machine.

**Cost-annotated evaluation / abort** — A variant big-step judgment `e ⇓ⁿ v` (or `e ⇓ⁿ ○` for an aborted partial derivation of size `n`) that records cost and can distinguish stuck from diverging computations, enabling a type-safety statement equivalent to progress and preservation.

**PCF** — A partial functional language (natural numbers with `z`, `s`, `ifz`) whose recursive functions `fun{τ1;τ2}(f.x.e)` provide general recursion; unlike System T it is not normalizing.

**Abstract machine** — A transition-system dynamics whose states are structured objects (not bare expressions); used to model control precisely.

**K machine / control stacks** — An abstract machine for call-by-value PCF whose states pair a control stack with an expression (`K ▷ e` evaluating `e`, `K ◁ v` returning `v` to `K`), avoiding the search rules of structural dynamics.

**Frame** — An element of a control stack representing a pending computation with an input and output type (`f : τ1 ↝ τ2`), e.g. `ap(−; e2)` or `s(−)`; stacks are typed by matching consecutive frame types.

**Continuations and exceptions** — Control features that control stacks make easy to express, motivating the K machine.

**Stuck state** — A non-final machine state with no transition (e.g. a stack frame incompatible with the value it receives); ruled out by well-typing via progress and preservation for states.

**Observational (contextual) equality** — Two programs are equal if interchangeable in every program context of base type without changing the observed numeric result; `e ≅ e′`.

**Logical equality** — An inductively-defined (on type structure) notion of program equality, split in a call-by-value setting into a value relation `v ∼ v′ : τ` and an expression relation `e ≈ e′ : τ`; designed to coincide with observational equality.

**Fundamental theorem (of a logical relation)** — The theorem a logical relation is designed to satisfy; here, that logical equality coincides with observational equality.

**Parametricity** — Reynolds's relational property that a polymorphic function preserves relations: applied to related arguments at related types it yields related results; the parametricity theorem states every well-typed expression `e : τ` satisfies `e ≈ e ∈ [τ]`.

**Relational interpretation of types** — The device, solving impredicativity in the logical relation, of interpreting a type variable by an arbitrary relation `R : τ ↔ τ′` between values, and defining `∀`-equality by quantifying over all such relations.

**Characterization by type ("free" theorems)** — A use of parametricity to determine a function's behavior from its type alone, e.g. proving every closed `f : ∀(t. t → t)` equals the polymorphic identity, or deducing `f [nat](0)(17) ⟼* 0` from behavior at `bool` for `f : ∀(t. t → t → t)`.

**Data abstraction and modularity** — A consequence of parametricity: a client parametric in a hidden implementation type cannot distinguish implementations, so a library may replace its representation without breaking clients.

**Substructural type system** — A type system that controls variable use by restricting the structural rules (weakening, contraction, exchange); corresponds to substructural logics such as linear logic, with uses in memory management, access control, concurrency, and resource analysis.

**Ordered type system** — A substructural system lacking exchange, where the order in which variables are introduced and used matters.

**Linear type system** — A substructural system with neither weakening nor contraction: every variable is used exactly once (variables treated as resources); its rules split the context between subterms.

**Affine type system** — A substructural system with weakening but not contraction: every variable is used at most once.

**Relevant type system** — A substructural system with contraction but not weakening: every variable is used at least once; affine and relevant systems are incomparable.

**Sharing judgment** — `Γ ▷ (Γ1, Γ2)`, distributing a context's variables to subterms (to left, right, or both), used to give a syntax-directed relevant type system.

**Share form** — An explicit syntactic form `share(e1; x1,x2.e2)` (a let-binding duplicating a value into two variables) that makes contraction visible in the syntax.

**Simply-typed λ-calculus with unit** — The base language used to present substructural systems: variables, abstraction, application, plus the unit type `1` with value `triv` (`⟨⟩`).
